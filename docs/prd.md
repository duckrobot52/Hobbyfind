# 제품 요구사항 정의서 (PRD): HobbyFind

---

## 1. 제품 상세 설명 (Detailed Product Description)

### 1.1. 서비스 설명

**HobbyFind**는 사용자가 복잡한 탐색 과정 없이 자신의 성향에 맞는 취미를 쉽고 직관적으로 찾을 수 있도록 돕는 취미 큐레이션 웹서비스입니다. 전체 취미를 '운동형', '지능형', '예술형' 3가지 핵심 카테고리로 대분류하여 제공하며, 사용자는 자신의 관심사에 따라 취미를 필터링하고 저장할 수 있습니다.

### 1.2. 핵심 목표

* **직관적인 탐색 환경 제공**: 비회원도 로그인 없이 웹사이트 인입 즉시 모든 취미 리스트를 대분류 필터를 통해 한눈에 확인할 수 있도록 유도합니다.
* **개인화된 취미 관리**: 회원가입한 유저에게 북마크 기능을 제공하여 선호하는 취미를 저장하고, 마이페이지에서 자신의 취미 성향 분포를 시각적 통계로 파악할 수 있게 합니다.
* **실습 목적의 정적 데이터 활용**: 고정된 18개의 취미 데이터를 기반으로 명확하고 일관된 데이터 바인딩 환경을 구축합니다.

## 3. 핵심 기능 및 사양 (Core Features and Specifications)

### 3.1. 고정 취미 데이터 세트 (Hobby Data Set)

> **비즈니스 규칙**: 시스템 내 취미 리스트는 아래 18개로 고정되며, 어떠한 경우에도 임의로 변경하거나 확장하지 않습니다.

* **운동형**: 조깅/러닝, 요가, 수영, 자전거, 클라이밍, 댄스
* **지능형**: 독서, 퍼즐, 체스, 프로그래밍, 외국어 학습, 사진 촬영
* **예술형**: 그림 그리기, 악기 연주, 요리, 서예, 도자기 만들기, 정원 가꾸기

## 4. 사용자 페르소나 및 시나리오 (User Persona and Scenarios)

### 4.1. 페르소나

* **페르소나 A (비회원 탐색 유저)**
* **이름/나이**: 김미정 (23세, 대학생)
* **특징**: 최근 여가 시간이 늘어나 새로운 활동을 해보고 싶지만 무엇을 해야 할지 모르는 상태입니다. 회원가입 없이 가볍게 취미 종류들을 둘러보고 싶어 합니다.


* **페르소나 B (회원 가입 유저)**
* **이름/나이**: 박수민 (31세, 직장인)
* **특징**: 평소 무기력함을 느껴 생산적인 취미를 계획 중입니다. 마음에 드는 취미 후보군들을 저장해 두고, 자신이 어떤 성향(운동형 vs 지능형 vs 예술형)에 더 치우쳐 있는지 시각적으로 확인하고 싶어 합니다.



### 4.2. 유저 시나리오 (User Scenarios)

* **시나리오 A (비회원의 카테고리 탐색)**
1. 사용자가 HobbyFind 루트 페이지에 접속하여 Hero 섹션의 소개 문구를 확인합니다.
2. 메인 콘텐츠 영역의 그리드에서 18개의 전체 취미 카드를 스크롤하며 둘러봅니다.
3. 상단 바 내비게이션에서 '예술형' 카테고리 필터를 클릭합니다.
4. 예술형 전용 페이지로 이동하며 그림 그리기, 악기 연주 등 예술형에 속한 6개의 취미 카드만 노출되는 화면을 탐색합니다.


* **시나리오 B (회원의 북마크 및 마이페이지 통계 확인)**
1. 직장인 박수민 유저가 아이디와 비밀번호를 입력하여 로그인을 완료합니다.
2. 홈 화면 그리드에서 '프로필', '외국어 학습'(지능형) 카드와 '클라이밍'(운동형) 카드의 북마크 버튼을 클릭합니다.
3. 상단 바의 마이페이지 메뉴로 진입합니다.
4. 자신이 북마크한 3개의 취미 리스트가 정상 노출되는 것을 확인합니다.
5. 상단 통계 차트 영역에서 자신의 취미 분포도가 지능형 66%, 운동형 33% 비율의 그래프로 시각화된 리포트를 확인합니다.

## 5. 기술 스택 추천 (Technical Stack Recommendations)

* **프레임워크 (Framework)**: **Next.js**
* **선정 사유**:
* 루트 페이지 및 카테고리별 페이지(`/category/[type]`)를 SSR(서버 사이드 렌더링) 또는 SSG(정적 사이드 생성)로 처리하여 초기 로딩 속도를 극대화하고 SEO를 최적화할 수 있습니다.
* Next.js의 내장 App Router를 활용하여 상단 바(Topbar) 레이아웃을 전역 공유하고 페이지 전환을 매끄럽게 구현합니다.




* **데이터 관리 및 저장소 (Storage)**: **관계형 데이터베이스 (RDB)**
* **선정 사유**: 유저 테이블과 취미 고정 메타데이터 테이블, 그리고 유저별 북마크 매핑 테이블 간의 관계(1:N, N:M)를 명확히 정의하고 데이터 정합성을 유지하기에 적합합니다.

## 6. 제한사항 (Limitations)

* **데이터 확장 불가**: 데이터베이스 및 UI에 바인딩되는 취미 목록은 제시된 18개로 완전히 고정되며, 관리자 페이지를 통한 추가, 수정, 삭제(CRUD) 기능은 시스템 스코프에서 원천 제외됩니다.
* **비회원 기능 제한**: 비회원은 북마크 등록 시도 시 데이터베이스 적재 API 호출이 차단되며, 반드시 클라이언트 단에서 가로채어 로그인 페이지로 리다이렉트하거나 경고 모달을 띄워야 합니다.
* **플랫폼 범위**: 본 서비스는 전적으로 웹 브라우저 최적화 웹서비스로 빌드되며, 별도의 네이티브 모바일 앱(iOS, Android) 패키징은 고려하지 않습니다.
