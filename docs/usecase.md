# 유스케이스 정의서 (Use Case Document): HobbyFind

본 문서는 취미 큐레이션 웹서비스 **HobbyFind**의 주요 기능에 대한 유스케이스를 정의합니다. 실제 시스템 설계 및 검증(QA)에 즉시 활용할 수 있도록 작성되었습니다.

---

## 1. 액터 정의 (Actor Definitions)

| 액터명 | 구분 | 설명 및 권한 범위 |
| --- | --- | --- |
| **비회원 사용자** | 주액터 | • 로그인하지 않은 방문자<br>

<br>• 홈 페이지 및 카테고리별 페이지 탐색 기능만 이용 가능<br>

<br>• 북마크 등록 및 마이페이지 접근 권한 없음 |
| **회원 사용자** | 주액터 | • 아이디/비밀번호 기반으로 인증을 완료한 유저<br>

<br>• 비회원의 모든 탐색 기능과 더불어 북마크 추가/해제, 마이페이지 조회 권한 보유 |
| **시스템** | 부액터 | • 취미 목록 데이터 제공 및 필터링 처리<br>

<br>• 인증 토큰(세션) 검증, 북마크 데이터의 생성/삭제 처리 및 카테고리별 분포 통계 연산 수행 |

---

## 2. 비즈니스 규칙 및 제약사항 (Business Rules and Constraints)

* **BR-01 (고정 데이터 제약)**: 시스템 내 취미 목록은 제공된 18개(운동형 6개, 지능형 6개, 예술형 6개)로 완전히 고정되며, 런타임 중에 추가되거나 변경되지 않습니다.
* **BR-02 (권한 제약)**: 마이페이지(`/mypage`)는 인증된 토큰을 보유한 회원 사용자만 접근할 수 있으며, 비회원이 URL 주소로 직접 접근을 시도할 경우 백엔드/미들웨어 단에서 원천 차단하고 로그인 페이지로 리다이렉트해야 합니다.
* **BR-03 (북마크 소유권)**: 북마크 데이터는 사용자 계정별로 독립적으로 매핑되어야 하며, 타인의 마이페이지 데이터나 북마크 리스트를 조회하는 행위는 불가합니다.

## 3. 상세 유스케이스 시나리오 및 흐름 (Detailed Use Case Scenarios)

### UC-01: 메인페이지 탐색 및 카테고리 필터링

#### A. 사전조건 및 사후조건

* **사전조건**: 사용자가 웹 브라우저를 통해 루트 URL(`/`)에 접속합니다.
* **사후조건**: 사용자의 화면에 선택한 카테고리에 맞는 취미 카드 그리드가 올바르게 렌더링됩니다.

#### B. 주요 단계 및 이벤트 흐름 (Main Steps)

1. **[사용자]** HobbyFind 웹서비스의 루트 페이지(`/`)에 인입합니다.
2. **[시스템]** 사이트 소개 문구가 포함된 Hero 섹션과 18개의 고정 취미 카드가 나열된 그리드 화면을 렌더링합니다.
3. **[사용자]** 메인 콘텐츠 영역의 대분류 필터(토글 스위치)에서 특정 카테고리(예: 운동형)를 선택합니다.
4. **[시스템]** 18개의 전체 취미 중 '운동형' 태그를 가진 6개의 취미 카드만 화면에 실시간 필터링하여 출력합니다.

#### C. 대체 흐름 및 엣지 케이스 (Alternative Flows)

* **대체 흐름 (카테고리 선택 이동)**: 사용자가 메인 콘텐츠 영역의 토글 대신 최상단 Top Bar의 카테고리 메뉴를 클릭하는 경우, 시스템은 전용 페이지 경로(`/category/[type]`)로 화면을 전환하고 해당 카테고리명과 소개 문구, 6개의 취미 카드 그리드를 노출합니다.

---

### UC-02: 회원가입 및 로그인

#### A. 사전조건 및 사후조건

* **사전조건**: 사용자가 로그인하지 않은 상태(비회원)여야 하며, 상단 바의 '로그인' 또는 '회원가입' 버튼을 통해 해당 라우트에 진입합니다.
* **사후조건**: 인증이 완료되어 세션/토큰이 발급되고, 상단 바가 회원용 레이아웃으로 갱신되며 홈 페이지로 이동합니다.

#### B. 주요 단계 및 이벤트 흐름 (Main Steps)

1. **[사용자]** Top Bar의 '로그인' 버튼을 클릭하여 `/login` 페이지로 진입합니다. (계정이 없다면 '회원가입' 링크를 클릭하여 `/signup` 페이지에서 아이디, 비밀번호 입력 및 약관 동의 후 계정을 생성합니다.)
2. **[사용자]** 아이디와 비밀번호를 입력 필드에 작성하고 제출 버튼을 클릭합니다.
3. **[시스템]** 수신된 아이디와 비밀번호를 데이터베이스 내부 정보와 대조하여 일치 여부를 검증합니다.
4. **[시스템]** 인증 성공 시 세션 토큰을 클라이언트에 발급하고, 유저를 루트 페이지(`/`)로 리다이렉트합니다. Top Bar는 [마이페이지/로그아웃] 상태로 전환됩니다.

#### C. 예외 처리 절차 (Exception Handling)

* **EX-01 (로그인 인증 실패)**: 사용자가 잘못된 아이디 혹은 비밀번호를 입력하고 제출한 경우, 시스템은 화면 전체를 새로고침하지 않고 폼 하단 영역에 붉은색 텍스트로 "아이디 또는 비밀번호가 일치하지 않습니다."라는 에러 메시지를 동적으로 표기합니다.
* **EX-02 (회원가입 데이터 누락)**: 회원가입 시 필수 필드가 비어있거나 약관 동의 체크박스가 누락된 경우, 시스템은 제출을 차단하고 해당 입력 필드 바로 밑에 경고 문구를 안내합니다.

---

### UC-03: 취미 북마크 추가 및 해제

#### A. 사전조건 및 사후조건

* **사전조건**: 사용자가 탐색 페이지(홈 또는 카테고리 페이지)에서 취미 카드를 열람하고 있는 상태여야 합니다.
* **사후조건**: 데이터베이스의 해당 유저 북마크 테이블에 데이터가 추가(Insert)되거나 삭제(Delete)되며, UI 상의 북마크 아이콘 상태가 반전됩니다.

#### B. 주요 단계 및 이벤트 흐름 (Main Steps)

1. **[사용자]** 렌더링된 취미 카드 내부에 위치한 북마크 아이콘 버튼을 클릭합니다.
2. **[시스템]** 현재 로그인된 세션 상태를 확인하여 회원 여부를 검증합니다.
3. **[시스템]** 인증된 회원이면, 해당 `유저 ID`와 `취미 ID` 쌍이 기존 데이터베이스에 존재하는지 확인합니다.
4. **[시스템]** 데이터가 존재하지 않는 경우 신규 북마크를 등록(활성화 색상 반영)하고, 존재하는 경우 해당 데이터를 데이터베이스에서 삭제(비활성화 색상 반영)하여 토글 상태를 업데이트합니다.

#### C. 대체 흐름 및 예외 처리 (Alternative Flows & Exception)

* **EX-03 (비회원의 북마크 시도)**: 2번 단계에서 로그인하지 않은 비회원 액터로 판명될 경우, 시스템은 데이터베이스 반영을 차단합니다. 화면에 "로그인이 필요한 기능입니다. 로그인 페이지로 이동하시겠습니까?"라는 컨디셔널 팝업 모달을 띄우고, 사용자가 승인하면 로그인 페이지(`/login`)로 즉시 이동시킵니다.

---

### UC-04: 마이페이지 조회

#### A. 사전조건 및 사후조건

* **사전조건**: 회원 사용자가 로그인 인증을 완료한 유효 세션 상태여야 합니다.
* **사후조건**: 사용자의 마이페이지 화면에 북마크 총 개수, 통계 그래프, 북마크한 취미 카드 리스트가 올바르게 로드됩니다.

#### B. 주요 단계 및 이벤트 흐름 (Main Steps)

1. **[사용자]** Top Bar 우측의 '마이페이지' 링크 버튼을 클릭하여 `/mypage` 라우트에 진입합니다.
2. **[시스템]** 로그인된 유저 ID와 연동된 북마크 테이블 데이터를 호출합니다.
3. **[시스템]** 조회된 전체 북마크 레코드의 총 개수를 연산하여 대시보드 상단에 배치합니다.
4. **[시스템]** 북마크된 취미들의 카테고리(운동형, 지능형, 예술형) 속성을 분류하여 각각의 백분율 비율을 계산한 뒤, 통계 차트 컴포넌트에 데이터를 바인딩하여 렌더링합니다.
5. **[시스템]** 하단 저장 목록 섹션에 북마크된 취미 카드들만 그리드 레이아웃 형태로 정렬하여 표기합니다.

#### C. 예외 처리 절차 (Exception Handling)

* **EX-04 (북마크 데이터가 없는 신규 유저)**: 마이페이지 진입 시 북마크 내역이 0개인 경우, 시스템은 통계 차트 영역에 비어있는 상태의 UI(Zero State)를 출력하고, 하단 그리드 영역에는 "아직 북마크한 취미가 없습니다. 마음에 드는 취미를 저장해 보세요!"라는 한글 안내 문구를 표시합니다.

## 4. 데이터 요구사항 및 데이터 흐름 (Data Requirements & Data Flow)

### 4.1. 데이터 흐름도 (Data Flow)

Next.js 환경에서 북마크 조작 시 발생하는 엔드-투-엔드 데이터 흐름의 구조입니다.

```
[회원 브라우저 (UI)] ────(취미 ID + 토크 액션)────> [Next.js API 엔드포인트]
                                                          │
                                                (JWT/세션 유효성 인증)
                                                          ▼
[회원 브라우저 (UI)] <───(성공 피드백 / 통계 반영)─── [관계형 데이터베이스 (RDB)]

```

### 4.2. 페이지별 요구 데이터

* **홈 / 카테고리**: 고정된 18개 취미 메타데이터(취미명, 이미지 경로, 대분류 카테고리 태그).
* **마이페이지**: 유저 식별자 데이터, 유저 매핑 북마크 레코드 배열, 카테고리별 합산 통계값 수치.

## 5. UI 및 보안 고려사항

### 5.1. 사용자 인터페이스 고려사항 (User Interface Considerations)

* **레이아웃 안정성**: 모바일 및 데스크톱 브라우저 환경에 모두 대응할 수 있도록 그리드 레이아웃에 반응형 중단점(Breakpoints)을 적용하며, 텍스트가 줄바꿈되더라도 카드의 물리적 높이가 깨지지 않도록 균등 배치 기법을 사용합니다.
* **접근성**: 입력 에러 피드백이나 비회원 차단 안내 팝업 메시지는 기획 규칙에 의거하여 번역투 문장을 배제하고 직관적인 한국어 명사형/구어체 구조로 일관되게 제공합니다.

### 5.2. 보안 및 개인정보 고려사항 (Security and Privacy Considerations)

* **인증 앤 인가 (Auth & Security)**: 회원 비밀번호는 데이터베이스 저장 시 반드시 단방향 해시 알고리즘(예: bcrypt)을 이용하여 암호화 가공 후 적재해야 합니다.
* **API 엔드포인트 보호**: 북마크 추가/삭제 및 마이페이지 데이터 요청 API 라우트는 요청 헤더의 세션 토큰 존재 여부를 검증하는 미들웨어 서킷을 통과해야만 백엔드 데이터에 접근할 수 있도록 설계합니다.
